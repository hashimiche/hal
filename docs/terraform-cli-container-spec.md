# HAL Terraform CLI Container Spec

## Context

The local Terraform Enterprise instance runs with a self-signed certificate.
That works fine in a browser, but `terraform login` and other Go-based clients fail when the certificate authority is not present in the host trust store.

For HAL, the goal is not to fix the host trust store globally. The goal is to provide an ephemeral, local-lab-friendly CLI container that already trusts the local TFE certificate and can run on the shared `hal-net` network.

## Problem Statement

We need a HAL workflow that lets a user run `terraform` and `tfx` against the local TFE instance without requiring host-level certificate trust changes.

We also need authentication to be bootstrapped automatically in the helper environment so users can start immediately without manual token file setup.

## Proposed User Experience

### Primary command

Add a new Terraform subcommand:

```bash
hal terraform cli
```

Short alias remains available through the existing Terraform alias:

```bash
hal tf cli
```

### Lifecycle actions

Build or refresh the helper image:

```bash
hal tf cli enable
```

Boot the helper container and open an interactive shell:

```bash
hal tf cli --console
hal tf cli -c
```

Destroy the helper container and clean up HAL-managed scenario workspaces:

```bash
hal tf cli disable --force
```

Expected behavior:

- `enable` builds or refreshes the local helper image used for CLI sessions.
- `--console` starts `hal-tfe-cli` on `hal-net`, then opens a shell in it.
- leaving the shell with `CTRL+D` or `exit` must not destroy the helper container
- after leaving the shell, the user can re-enter later with `hal tf cli -c` without a rebuild/bootstrap cycle
- If `--console` is used and the image does not exist yet, HAL should build it first.
- `disable` requests confirmation in interactive mode, then removes the helper container and HAL-managed scenario workspaces from TFE.
- `disable --force` performs the same teardown without prompting.
- The container is intentionally disposable and optimized for local demos, tests, and proof-of-concept work.

### Console UX

When entering console mode with `hal tf cli -c`, print a clear welcome banner before dropping into the shell.

The banner should include:

- HAL ASCII art (or a compact HAL-branded header)
- explanation that the user is in an ephemeral TFE helper environment
- example commands (`terraform`, `tfx`, `git`, `curl`)
- reminder that `terraform login` is not needed because auth is already bootstrapped
- reminder that `exit` only leaves the shell and `hal tf cli -c` re-enters the same helper container
- explicit exit guidance (`CTRL+D` or `exit`)

### Build UX

During `hal tf cli enable`, HAL should default to a compact progress animation rather than streaming full Docker build logs.

If detailed troubleshooting is needed, users can opt into raw build output with a verbose flag.

The current animation uses a progressive Terraform-style ASCII logo reveal with a `#` progress bar.

## Container Requirements

The helper image must include:

- base image: `ghcr.io/straubt1/tfx:latest`
- `terraform`
- `tfx` (provided by base image)
- `git`
- `curl` (if not already present in base image)
- CA trust configured for the local TFE certificate
- a usable shell such as `sh` or `bash`

The running container must:

- join `hal-net`
- be named `hal-tfe-cli`
- trust the certificate generated by HAL for `tfe.localhost`
- be able to resolve and reach the local TFE endpoint

## Certificate Strategy

HAL already generates the local TFE certificate material during `hal terraform create` under the HAL user state directory.

The CLI container should reuse that certificate rather than generating a second trust chain.

Baseline approach:

- mount the HAL-generated certificate into the container read-only
- copy it into the container CA trust path during startup or image build
- refresh the container trust store before exposing the shell to the user

Because the image is built locally by HAL, certificate trust setup can be handled during image build and validated again at runtime before opening console mode.

Reference certificate location today:

```text
~/.hal/tfe-certs/cert.pem
```

## Connectivity Assumptions

The container should target the local TFE endpoint already exposed by HAL:

```text
https://tfe.localhost:8443
```

Recommended default environment variables inside the helper container:

```bash
TFE_HOST=https://tfe.localhost:8443
TFE_DEFAULT_ORGANIZATION=hal
```

Optional follow-up:

- expose additional helper environment variables for advanced workflows

## Token Bootstrap And Auth Files

Before opening the interactive shell in `hal tf cli -c`, HAL should create or refresh a TFE user token and write authentication files inside the helper container.

### TFX configuration

Write `~/.tfx.hcl` with:

```hcl
hostname            = "tfe.localhost:8443"
defaultOrganization = "hal"
token               = "<generated-user-token>"
```

### Terraform CLI configuration

Create `~/.terraform.d/credentials.tfrc.json` with a matching token entry:

```json
{
	"credentials": {
		"tfe.localhost:8443": {
			"token": "<generated-user-token>"
		}
	}
}
```

This ensures Terraform CLI is already authenticated for the local TFE hostname before the user runs any command.

### Token lifecycle expectations

- token should be created automatically by HAL for the helper session
- token bootstrap should authenticate with the default local admin credentials `haladmin/hal9000FTW`
- token should be scoped for local lab usage and not reused outside this flow
- `update` should rotate token material by recreating auth files
- token values must never be printed in command output
- helper token/bootstrap no longer requires creating a default TFE project name

## Repo Workflow Inside The Container

The intended workflow is:

1. Build or refresh the helper image with `hal tf cli enable`.
2. Start and enter the helper container with `hal tf cli -c`.
3. Clone a Git repository inside the container.
4. Run `terraform`, `terraform login`, `tfx`, or related commands from that cloned repository.

This stays aligned with HAL's local-lab model:

- ephemeral
- isolated
- no host trust-store mutation
- suitable for demos and quick experiments

## Scenario Seeding

HAL-managed demo scenarios may create multiple workspaces and matching repos for exploration inside the helper container.

For the current dad-joke scenario set:

- repos live under `/workspaces/<repo>` inside `hal-tfe-cli`
- `hal tf cli -c` ensures scenario workspaces exist in TFE and are assigned across projects `Dave` and `Frank`
- `hal tf cli -c` auto-seeds the default `/workspaces` scenario directories (with `main.tf`) when they are missing
- by default, `hal tf cli` no longer creates a `HAL-CLI` project; `--tfe-project` is optional
- `disable` cleanup removes HAL-managed scenario workspaces from TFE and deletes the helper container

Current default workspace mapping:

- `hal-lucinated` -> `Dave`
- `hal-lelujah` -> `Dave`
- `hal-ibut` -> `Dave`
- `hal-ogen` -> `Frank`
- `hal-oween` -> `Frank`

Default run seeding mix (first helper bootstrap on a fresh container):

- `hal-lucinated`: one plan-only run, then one apply run
- `hal-lelujah`: one plan-only run, then one apply run
- `hal-ogen`: plan-only run
- `hal-oween`: plan-only run
- `hal-ibut`: no activity (intentionally empty run history)

Run seeding is launched in the background after `hal tf cli -c` bootstrap so shell entry is faster.

## Suggested Command Semantics

When `hal tf cli` is run with no flags, prefer a lightweight status view.

Suggested output:

- whether the local helper image exists
- whether `hal-tfe-cli` exists
- whether it is running
- whether `hal-tfe` is running
- the target TFE URL
- the next suggested command

Example guidance:

```text
Run 'hal tf cli enable' to build the helper image.
Run 'hal tf cli -c' to start a session.
```

## Implementation Notes

Recommended implementation shape:

- new file: `cmd/terraform/cli.go`
- register a new `cliCmd` under `cmd/terraform/terraform.go`
- follow existing HAL feature-command patterns used by `workspace` and Vault feature flows
- use `global.DetectEngine()` and `global.EnsureNetwork()`
- keep the container name stable as `hal-tfe-cli`

Runtime behaviors worth keeping explicit:

- fail early if `hal-tfe` is not running
- fail clearly if the TFE certificate does not exist yet
- support `update` (or `--force` compatibility) to rebuild the image and recreate the helper container cleanly
- prompt for interactive confirmation on `disable`
- allow `disable --force` as a non-interactive confirmation bypass
- keep image build and container startup idempotent

## Decisions Captured

The following decisions are now explicit for the first implementation:

1. HAL builds the helper image locally, using `ghcr.io/straubt1/tfx:latest` as the base image.
2. `hal tf cli enable` builds/refreshes the image; `hal tf cli -c` starts the container and enters the session.
3. Cloning happens inside the container for V1 (no bind mount by default).
4. Scope for V1 includes certificate trust plus automatic user-token bootstrap for TFX and Terraform CLI.
5. The image and tooling policy for now is `latest` (no pinned version in V1).

## Optional Local Mount

The helper supports an optional host-directory bind mount for persistent local repos:

```bash
hal tf cli --console --local-directory /path/to/local/repos
```

Behavior:

- no mount by default (ephemeral container-only filesystem)
- when provided, the host path is mounted to `/workspaces` in the helper container
- `disable` removes container and managed TFE workspaces, but does not delete host files

## Acceptance Criteria

The first implementation is successful if all of the following are true:

1. `hal tf cli enable` builds or refreshes the local helper image.
2. `hal tf cli -c` starts `hal-tfe-cli` on `hal-net` and opens an interactive shell session.
3. Console mode shows a short welcome banner with usage hints and exit instructions.
4. Inside the container, `terraform version`, `tfx --help`, `git --version`, and `curl --version` work.
5. Inside the container, Go-based TLS clients trust the local TFE certificate.
6. HAL writes `~/.tfx.hcl` with hostname, default org (`hal`), and a generated token before the shell opens.
7. HAL writes `~/.terraform.d/credentials.tfrc.json` with a token for `tfe.localhost:8443` before the shell opens.
8. `terraform login https://tfe.localhost:8443` works directly in the helper environment.
9. The flow does not require adding the self-signed certificate to the macOS host trust store.
10. Leaving the shell with `exit` or `CTRL+D` preserves the helper container for later re-entry with `hal tf cli -c`.
11. `hal tf cli disable` prompts for confirmation in interactive mode before teardown.
12. `hal tf cli disable --force` performs non-interactive teardown.
13. `hal tf cli disable` teardown removes the helper container and deletes HAL-managed scenario workspaces.
14. `hal tf cli -c` ensures `Dave`/`Frank` projects and the default scenario workspaces exist in TFE after a fresh reset.

## Non-Goals For V1

- persistent workstation environment management
- full host filesystem integration
- general-purpose development container features unrelated to local TFE access
- solving every Terraform Enterprise authentication workflow in the first pass
